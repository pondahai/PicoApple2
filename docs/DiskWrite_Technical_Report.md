# 🍎 PicoApple2 技術專題：磁碟寫入（Disk Write）的解題與實作

在 Apple II 模擬器中，磁碟寫入是最具挑戰性的任務之一。它不僅涉及 CPU 的精確時序，還必須在資源極度受限的 RP2040 上，將虛擬的磁頭訊號轉化為實體 SD 卡上的檔案變動。

以下是我們在 **PicoApple2** 中解決此問題的五個關鍵階段：

---

### 第一階段：核心邏輯的「物理級」模擬 (Q7 Write Lock)

**問題**：早期版本中，當 Apple II 軟體試圖寫入資料時，磁軌資料經常被「污染」或寫入失敗。
**解法**：
*   **Q7 狀態機鎖定**：在 6502 Disk II 控制器中，Q7 暫存器控制寫入電路。我們實作了嚴格的物理邏輯：**只要 Q7 開啟（寫入模式），立即強制切斷讀取電路**。
*   **防止回饋干擾**：這防止了磁頭正在寫入的位元立即被 `data_latch` 讀回，從而避免了模擬器內部的邏輯衝突。

### 第二階段：SD 卡存取的「原地覆蓋」陷阱 (The "r+" Trap)

**問題**：使用 Arduino 標準的 `SD.open(path, FILE_WRITE)` 時，資料會被「附加」到檔案末端，而不是覆蓋原有的磁軌內容。
**解法**：
*   **模式切換**：改用 `"r+"` (Read/Write) 模式開啟 `.DSK` 檔案。
*   **精確尋址**：利用 `diskFile.seek(track * 4096)` 定位到特定的磁軌位址進行覆蓋，確保虛擬磁軌與實體檔案的位元組對齊。

### 第三階段：髒位元優先解碼 (Dirty-aware Denibblization)

**問題**：RP2040 的 RAM 只有 264KB，無法存放整張磁碟的原始位元流（Raw Nibbles）。
**解法**：
*   **Selective Sync**：我們不儲存原始位元，而是只在核心偵測到「髒位元」（Dirty Bit）時才觸發解碼。
*   **解碼過濾**：在 `apple2_get_denibblized_track` 過程中，利用 `dirty_mask` 辨識哪些扇區（Sector）被真正修改過，排除初始化產生的偽信號，確保只有有效資料會寫入 SD 卡。

### 第四階段：物理層級的強力驗證 (Physical Verification)

**問題**：SD 卡可能因為 SPI 雜訊或寫入延遲導致資料毀損。
**解法**：
*   **寫入後讀回**：在 `flushDirtyTrack()` 函數中，寫入完成後立即執行 `diskFile.close()` 刷新 Metadata，隨後**重新開啟檔案並讀回前 64 位元組進行比對**。
*   **偵錯日誌**：如果驗證失敗，系統會輸出 `SD VERIFY ERROR`，這讓我們在開發階段能快速揪出 SPI 時序問題。

### 第五階段：效能衝刺 (The "Disk Boost" Acceleration)

**問題**：磁碟編碼/解碼與 SD 卡 I/O 是沈重的運算負擔，會導致模擬速度掉到 1.023MHz 以下。
**解法**：
*   **超頻至 250MHz**：提昇 RP2040 的基礎運算能力。
*   **動態限速解除**：
    ```cpp
    if (!motor_on) {
        // 只有在馬達停止時才進行 1.023MHz 的人工限速
        delayMicroseconds(expected - actual);
    }
    ```
*   **成果**：當磁碟馬達運轉時，Pico 會「全速奔跑」，大幅縮短了存檔與載入的等待時間。

---

### 💡 總結與 Lessons Learned

這套解法成功的關鍵在於：**「尊重物理時序，但利用現代硬體性能來彌補模擬開銷」**。

1.  **物理性優先**：先解決 Q7 的讀寫鎖定，確保核心邏輯正確。
2.  **安全性次之**：實作 `"r+"` 與讀回驗證，確保 SD 卡資料安全。
3.  **效能最後優化**：透過超頻與動態調速，解決最後的流暢度問題。
